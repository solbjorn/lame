# Encoding-speed comparison {#maintainer_perf}

`maintainer/perf-compare.sh` answers one question about a change: is the
encoder faster, slower, or indistinguishable afterwards?

It encodes one input file with a baseline and a candidate build, times both,
and reports a confidence interval over the difference &mdash; together with
whether the two builds produced the same bitstream at all.

## Why two build matrices

The script does not build anything. It takes one cell out of each of two
matrices generated by `maintainer/gen-build-matrix.sh` (see @ref
maintainer_build_matrix): one built from the baseline source, one built from
the candidate source. Comparing a cell against the same cell keeps the
compiler and the configure options out of the difference, so that what is left
is the source change.

Timing a single tree before and after a rebuild would work too, but it makes
re-running the baseline a rebuild away, and any two of the comparisons in a
review are then hours apart on a machine whose state has moved on.

## Speed alone is not the result

Unless a change is meant to alter the output, a faster encoder that emits
different bits is a bug, not an optimization. The script therefore encodes once
with each build and compares the two files before it times anything. A
differing bitstream is reported next to the timings and makes the run fail; the
timings are still printed, because a rejected result is easier to act on when
one can see what it would have been.

## Usage

```
sh maintainer/perf-compare.sh -o DIR -n DIR -i FILE [-c CELL] [-e ARGS] [-r N]
sh maintainer/perf-compare.sh -o DIR -n DIR -l
```

| Option    | Meaning                                                             |
|-----------|---------------------------------------------------------------------|
| `-o DIR`  | baseline build matrix                                               |
| `-n DIR`  | candidate build matrix, laid out the same way                       |
| `-i FILE` | input audio; several minutes of real music, not a short clip        |
| `-c CELL` | cell to compare, e.g. `gcc/full` (default: `gcc/full`)              |
| `-e ARGS` | encoder arguments (default: `-V2`)                                  |
| `-r N`    | hot runs, at least 10 (default: 10)                                 |
| `-l`      | list the cells both matrices have in common, and exit               |

`-c` also takes a bare cell name: `-c static` resolves when only one compiler
provides that cell, and reports the candidates rather than guessing when
several do.

`-e` takes **all** the encoder arguments as its single value, split on the
spaces inside it &mdash; so more than one goes in one quoted argument:

```
sh maintainer/perf-compare.sh -o ../base-matrix -n ../cand-matrix \
   -i ~/audio/album.wav -e "-V0 -m j"
```

### Examples

List what can be compared, before picking a cell:

```
sh maintainer/perf-compare.sh -o ../base-matrix -n ../cand-matrix -l
```

The default comparison &mdash; `gcc/full`, `-V2`, ten hot runs:

```
sh maintainer/perf-compare.sh -o ../base-matrix -n ../cand-matrix \
   -i ~/audio/album.wav
```

A different compiler and configuration, at CBR, with more samples:

```
sh maintainer/perf-compare.sh -o ../base-matrix -n ../cand-matrix \
   -c clang/static -i ~/audio/album.wav -e "-b 192" -r 20
```

Pin the run to one core, which is the single most effective thing one can do
to the numbers:

```
taskset -c 2 sh maintainer/perf-compare.sh ...     # Linux
cpuset -l 1  sh maintainer/perf-compare.sh ...     # FreeBSD
```

### Comparing build flags instead of a source change

The two matrices do not have to come from two different sources. Comparing what
a build option does to the encoding speed is the same operation with the source
held fixed instead of the flags.

`--enable-native` has no cell of its own, because a native-tuned binary is not
meant to be redistributed and the matrix is there to test what is. A cell is
recognized by its `config.status`, though, so one built by hand in the same
`<matrix>/<compiler>/<cell>` layout is compared like any other:

```
mkdir -p ~/base-matrix/gcc/full ~/native-matrix/gcc/full

cd ~/base-matrix/gcc/full
CC=gcc ~/lame/configure --enable-dynamic-frontends && make -j16

cd ~/native-matrix/gcc/full
CC=gcc ~/lame/configure --enable-dynamic-frontends --enable-native && make -j16

taskset -c 2 sh maintainer/perf-compare.sh \
   -o ~/base-matrix -n ~/native-matrix -i ~/audio/music.wav
```

## Prerequisites

**hyperfine** runs and times the encoder; it is required.

**ministat** turns the two sets of hot runs into a confidence interval. Where
it is missing the script says so, reports hyperfine's own comparison instead,
and the run stands.

```
apt install hyperfine ministat                          # Debian/Ubuntu
pkg install hyperfine                                   # FreeBSD (ministat is in base)
pacman -S mingw-w64-ucrt-x86_64-hyperfine               # MSYS2 (UCRT64)
```

There is no native-Windows counterpart to this script. Comparing timings needs
the two builds to differ in nothing but the source, and the POSIX matrix is
where that holds.

### The sign

Every number in the report is a **time**, so it moves the opposite way to the
speed. The difference is the candidate minus the baseline:

- **negative &mdash; the candidate is faster.** It took less time. This is the
  result one is usually hoping for, and it is the minus sign that says so.
- **positive &mdash; the candidate is slower.** It took more time.

Here `-7.54699%` means the native build encodes this file 7.5% faster than the
default one, not 7.5% worse.

`ministat` marks the baseline `x` and the candidate `+`, both in the legend and
in the plot. The plot is the two sets of runs on one axis: `A` is the average,
`M` the median where it is far enough from the average to be drawn separately,
and the bracket around them the confidence interval. Two brackets that do not
overlap are the picture of the difference the numbers below state.

### The two "+/-" are not the same thing

They look alike and mean different things:

- `2.195 s +/- 0.016` on the per-build lines is a **standard deviation**: how
  much the ten runs of *that* build scattered.
- `-0.165623 +/- 0.0123796` under `Difference at 95.0% confidence` is the
  **half-width of the confidence interval** on the difference between the two
  builds. It says the true difference lies between -0.178 s and -0.153 s with
  95% confidence.

The second is the one to quote. The first only feeds the CV.

When there is nothing to report, ministat says so in as many words &mdash;
`No difference proven at 95.0% confidence`. That is a result: the change did not
move the encoding speed measurably on this machine with this input. It is not a
weaker form of "faster", and it is not "too few samples" either.

### The bitstream line, when the flags are what changed

The run above **fails** (exit 1) on `bitstream: DIFFERENT`, and here that is
expected rather than a bug. `--enable-native` adds `-march=native`, which on any
machine with FMA lets the compiler contract a multiply and an add into one
instruction that rounds once instead of twice. The arithmetic feeding the
psychoacoustic model is then not bit-identical, and the encoder makes different
decisions downstream: for a four-minute file, nearly every byte after the first
50 differs, and the file length changes by a few bytes.

Each build is still deterministic &mdash; it reproduces its own output exactly
&mdash; and both files are valid MP3.

The script cannot tell an expected output change from an accidental one, so it
reports the difference and fails either way. Deciding which one it is stays with
the person reading the report. When the flags are the variable, read the
bitstream line as information; when the source is the variable and the change
was not meant to alter the output, it is a bug and outranks the timings.

Whether a bitstream difference is *audible* is a separate question, and this
script cannot answer it. Scoring these same two builds against a reference
corpus (see @ref maintainer_quality) reports no perceptual difference at all: on
this change the bits move everywhere and the audio does not.

### The measurement

The coefficient of variation decides whether the run means anything, before any
difference between the two builds does:

| CV        | What it means                                                     |
|-----------|-------------------------------------------------------------------|
| &le; 2%   | trustworthy                                                       |
| 2% &ndash; 5% | borderline; re-run with `-r 20`                               |
| &gt; 5%   | the run is rejected and the script fails                          |

Above 5% more samples do not help: the cause is the machine, not the sample
count. Frequency scaling, turbo boost, thermal throttling, a power plan that is
not the performance one, or something else running are the usual ones. Pinning
to a core (`taskset`/`cpuset`) is worth trying first; it routinely takes the CV
from the high single digits to well under one percent.

## Using it to validate a patchset

1. Generate and build a matrix from the unpatched source.
2. Generate and build a second matrix from the patched source.
3. Compare the base cell, on a quiet machine, pinned to a core.
4. Check the bitstream line first. If it says `DIFFERENT` and the change was
   not meant to alter the output, stop &mdash; the timings are moot.
5. Repeat for any cell the change is specific to (a `--disable-decoder` change
   belongs measured in `nodecoder`, not only in `full`).

Report the confidence interval, the CV, and the input, not a single percentage:
a speedup on one file at one bitrate is a speedup on one file at one bitrate.
